Push vs Pull

Two V3 traversal strategies

PULLPUSH
OwnerDestination nodeSource node
ReadsIncoming neighbors-
WritesOwn rank, no conflictOutgoing neighbors, atomic
SyncWarp shuffleatomicAdd
Power-law perfHub nodes read, not writtenHub nodes can cause atomic contention
BottleneckMemory bandwidthAtomic serialization on hubs

Pull usually wins on power-law graphs because high-degree destinations are read by one owning thread or warp instead of being targeted by many writers. That removes atomic conflicts from the hottest rank locations and makes the main cost a predictable gather over incoming CSR. Push can still win when outgoing traversal locality and lower contention outweigh those atomic costs.

Verification Checklist

Detected evidence and manual checks

1

Problem

PageRank on billion-edge graphs spends most runtime in sparse matrix-vector style traversal. CPU execution is easy to verify but too slow for live iteration. GPU kernels help by exposing parallelism across graph rows, edges, and rank updates.

2

Approach

The project stores graphs as CSR, then moves from CPU baseline to V1 SpMV, V2 fused update, and V3 pull/push kernels. Pull gathers incoming neighbors; push scatters from source nodes.

3

Correctness

SciPy reference PageRank anchors validation. The dashboard reports L1 error and Spearman values from the benchmark CSV only, with no generated numbers.
